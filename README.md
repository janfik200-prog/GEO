# Прогноз золото-урановых рудных узлов (Анабарский щит, лист R-48-XI, XII)

Задача проекта — разработать метод прогнозирования золото-урановых рудных узлов
с использованием машинного обучения. **Гипотеза:** ML-подход даёт более
качественный прогноз, чем классический критериальный анализ (метод ГИС Интегро
«Таксономия по критериям»), так как способен учитывать сложные нелинейные
сочетания геологических факторов.

Данные: ФГБУ «ВНИГНИ», Отделение Геоинформатики (ГИС Интегро).

## Что показано

- **На Анабаре (63 метки Au-U) гипотезу строго доказать нельзя** — меток слишком
  мало. Раннее «ML в разы лучше» оказалось циркулярным (ML мерили на собственных
  обучающих точках). Урок зафиксирован: качество мерить только на held-out
  реальных точках (пространственная блочная CV, lift@N%), никогда на псевдометках.
- **Доказательство перенесено на сторонние датасеты с тысячами меток:**
  - **CMMI/Lawley (USGS, Zn-Pb, США+Канада+Австралия):** ML значимо лучше
    критериального — ΔAUC +0.08…+0.11 (США+Канада), +0.15 (Австралия);
    бутстрэп-ДИ не включает 0.
  - **NURE + MRDS (Au/U, США-Запад):** признаки коммодити-специфичны —
    радиометрия NURE (eU/K/eTh) помогает урану (AUC 0.65→0.82) и бесполезна
    золоту; фиксированно-весовой критериальный это не умеет. ML побеждает для
    обоих типов оруденения.
- **Критериальный baseline = точная реконструкция метода ГИС Интегро**
  «Таксономия по критериям» (степенная трансформация → min-max → взвешенное
  L1-расстояние до эталона-минимума, мера Плюты). Веса восстановлены регрессией
  по нативному `prognoz.prognoz.property` (R²=1.000).

## Структура

```
src/           боевой Python-пакет (логика прогноза, единственный источник истины)
notebook/      forecast_main.ipynb (прогноз) + validation_report.ipynb (валидация)
tests/         pytest: юниты + smoke-тест пайплайна
experiments/   исследовательские прогоны (проверка гипотезы на сторонних датасетах,
               transfer-эксперименты); общий код — в experiments/common.py
cache_paths.py пути постоянного кэша скачанных данных (datacache/, env GIS_CACHE)
data/          входные данные ГИС Интегро (шейп-файлы, нативные расчёты)
outputs/       PNG-карты экспериментов · metrics/ сводные таблицы (оба gitignored)
```

`src/` — боевой пайплайн (ноутбуки и тесты). `experiments/` — исследовательские
скрипты, не входящие в боевой путь.

## Запуск

```bash
source .venv/bin/activate
pip install -r requirements.txt

# тесты
python3 -m pytest tests/ -q

# боевой прогноз и валидация
jupyter notebook notebook/forecast_main.ipynb
jupyter notebook notebook/validation_report.ipynb

# исследовательские прогоны (только из корня репозитория, как модули):
python3 -m experiments.cmmi_terrain             # ML vs критериальный (США+Канада)
python3 -m experiments.nure_uranium             # Au/U-проверка (NURE + MRDS)
python3 -m experiments.anabar_uranium_transfer  # перенос U-модели на сетку Анабара
```

## Данные и кэш

Внешние данные (геофизика GMT, радиометрия NURE, метки MRDS/CMMI) скачиваются
идемпотентно в постоянный кэш `datacache/` (не в `/tmp` — он чистится при
перезапуске сессии). Пути — через модуль `cache_paths.py`, переопределяются
переменной окружения `GIS_CACHE`. Кэш не публикуется в git (сотни МБ).

Метод (`compute_geo_score`, `experiments/`) переносится на любую территорию;
веса конкретной модели — нет (для другого типа оруденения транслируется метод,
а не обученные веса).
